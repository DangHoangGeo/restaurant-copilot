# Web Performance Improvement Plan (2025-08-09)

Owner: Web app (Next.js, React, TypeScript)
Scope: Admin dashboard, owner APIs, catalog APIs, global web perf/security

# A. Review validation (quick check)
- Issue 1 (Dynamic style injection): Confirmed. `web/app/[locale]/(restaurant)/dashboard/admin-layout-client.tsx` embeds `primaryColor` directly in a `<style>` tag. Risk: CSS/script injection. Fix: sanitize or whitelist color and avoid raw template injection.
- Issue 2 (Multiple dashboard fetches): Confirmed. `dashboard-client-content.tsx` fires 5 fetches on first paint. Fix: aggregate on server or via a single API.
- Issue 3 (No rate limiting/CSRF on POST): Confirmed in `web/app/api/v1/owner/orders/route.ts`. Fix: add per-route rate limit and CSRF/origin checks.
- Issue 4 (N+1 + no transaction in order creation): Confirmed. Multiple sequential queries per item; separate insert for order and items with manual rollback. Fix: bulk prefetch + Postgres function to handle transactional creation.
- Issue 5 (Unvalidated query params in orders GET): Confirmed. Free-form `fromDate`, `toDate`, `status`, `period`. Fix: zod validation, clamp ranges, pagination.
- Issue 6 (Unbounded categories): Confirmed. Full categories with nested menu items/sizes/toppings. Fix: pagination, selective includes, default limits.
- Issue 7 (Error detail leakage): Confirmed. Several routes return `error.message`. Fix: generic client errors + correlation IDs; log details server-side.

# B. High-impact initiatives (what/why/how)
1) Aggregate and cache dashboard data
- Why: Reduce TTFB and bandwidth; fewer waterfalls.
- How:
  - Create a single endpoint `GET /api/v1/owner/dashboard/aggregate` or move fetching to a Server Component with `fetch` on server and `revalidateTag("dashboard")`.
  - Cache read-heavy parts (popular items, sales-over-time) with `revalidate: 60-300` and tagging; fetch active/volatile metrics separately or add SWR on client for freshness.
  - Lazy-load heavy visuals (charts) using `next/dynamic` with `ssr: false`.

2) Transactional order creation (remove N+1 and partial writes)
- Why: Consistency and speed under load.
- How:
  - Bulk-prefetch reference data in one query each: all `menu_item_ids`, `menu_item_size_ids`, `topping_ids` used in the payload.
  - Create a Postgres function `create_order(p_restaurant uuid, p_table uuid, p_guest_count int, p_items jsonb)` that:
    - Validates ownership and availability.
    - Calculates prices.
    - Inserts into `orders` and `order_items` inside a single transaction, returns the new `order_id`.
  - Call via `supabaseAdmin.rpc('create_order', ...)`. Add migration and unit tests.

3) Pagination and selective fetch for catalog endpoints
- Why: Avoid huge payloads and long render times.
- How:
  - `GET /api/v1/owner/categories`: add `page`, `pageSize (<=50)`, and `include=items|sizes|toppings` parameters. Default to categories only with counts.
  - Return `{ data, pagination: { page, pageSize, total } }`. Add indexes to support ordering and filtering.

4) Robust input validation + safe error responses
- Why: Security and predictable performance.
- How:
  - Use zod schemas for all route query/body payloads (orders GET/POST, categories GET/POST).
  - Clamp date windows (e.g., max 31 days) and status enums; require pagination for list endpoints.
  - Replace `error.message` in responses with generic messages; include `requestId` and log details with the existing `logger`.

5) Rate limiting and CSRF
- Why: Protect state-changing endpoints and infra.
- How:
  - Rate limiting: Adopt `@upstash/ratelimit` + `@upstash/redis` (Edge-compatible) or equivalent Redis. Start with 10 req/min per user/IP for POST/PUT/PATCH/DELETE under `/api/v1/owner/**`.
  - CSRF: For cookie-auth endpoints, implement double-submit token or use `Origin`/`Referer` checks for same-site requests, plus SameSite=strict/lax cookies. Add a custom header `X-Owner-CSRF` checked by middleware on unsafe methods.

6) Server-centric rendering and code-splitting
- Why: Smaller JS bundles and faster hydration.
- How:
  - Move dashboard data fetching to Server Components where feasible; keep only interactive widgets as client components.
  - Dynamic import heavy modules (charts, tables) with `loading` skeletons; tree-shake icon libs (only import used icons).

7) Safe theming and CSS
- Why: Prevent CSS/script injection and layout thrash.
- How:
  - Replace template-injected `<style>` with: validate hex color via regex (`/^#([A-Fa-f0-9]{3}|[A-Fa-f0-9]{6})$/`), then set CSS variables via a safe `style` object on a wrapper or add a `data-theme-color` attribute consumed by a static stylesheet.
  - Fallback to a default theme color if validation fails. Add unit tests for sanitizer.

8) Caching, headers, and assets
- Why: Reduce network costs and speed up loads.
- How:
  - Set `Cache-Control` on API reads (short-lived, e.g., 30-120s) and static assets (1y immutable). Use Next.js Route Handlers’ `next: { revalidate }`.
  - Optimize images via `next/image`. Preload critical fonts; enable font-display: swap.
  - Consider enabling `brotli` compression (Next.js defaults OK) and HTTP keep-alive.

9) Database indexes and query plans
- Why: Keep queries fast under load.
- How: Add/verify indexes:
  - `orders(restaurant_id, status, created_at desc)`; `order_items(order_id)`.
  - `menu_item_sizes(menu_item_id)`; `toppings(menu_item_id)`.
  - For filtering by time/status, ensure composite indexes match where/order patterns.

10) Observability and perf budgets
- Why: Detect regressions.
- How:
  - Add log fields: `requestId`, latency, result size. Aggregate in your log store.
  - Add Lighthouse CI budget for key pages; track TTFB, LCP, CLS; set thresholds.

# C. Concrete tasks (sequenced)
## Phase 0 – Setup and guardrails
- T0.1 Add shared validation utilities in `web/lib/utils/validation.ts` (zod schemas for dates, enums, pagination). Tests in `web/lib/utils/__tests__/validation.test.ts`.
- T0.2 Implement safe color util `sanitizeHexColor` in `web/lib/utils/colors.ts`, with tests.
- T0.3 Introduce standard API error helper in `web/lib/server/apiError.ts` returning `{ code, message, requestId }` (no internals). Use `logger` for details.

## Phase 1 – Dashboard
- T1.1 Create `GET /api/v1/owner/dashboard/aggregate` aggregating metrics, recent orders, popular items, sales, low stock in one response. Tag caches and set `revalidate`.
- T1.2 Convert `dashboard-client-content.tsx` to fetch from the aggregate endpoint (or a Server Component wrapper). Dynamic-import charts with `loading`.
- T1.3 Tree-shake icons; only import used icons.

## Phase 2 – Orders API
- T2.1 Add zod schema for orders GET query; require pagination (limit<=100, cursor/offset). Clamp date range to 31 days; validate status enum.
- T2.2 Bulk-fetch sizes/toppings/menu_items for GET to avoid per-item follow-ups; limit nested fields; paginate orders list.
- T2.3 Add rate limiting and CSRF/origin checks for POST. Reject cross-site without valid token/header. Return generic errors with requestId.
- T2.4 Create SQL migration `infra/migrations/022_create_order_transaction.sql` with function `create_order(...)` that validates and inserts transactionally. Replace POST logic with `rpc` call. Add tests.

## Phase 3 – Categories API
- T3.1 Add query params: `page`, `pageSize (<=50)`, `include` with whitelist. Default to categories only.
- T3.2 Implement counts (e.g., `items_count`) via `select` or a view; only fetch nested data when requested.
- T3.3 Sanitize responses; remove `error.message` from outputs. Return `{ message: '...' }` and `requestId`.

## Phase 4 – Theming and security headers
- T4.1 Update `admin-layout-client.tsx` to use `sanitizeHexColor` and set CSS vars via a safe `style` prop or data attribute. Remove raw `<style>` template.
- T4.2 Add security headers/CSP in `middleware.ts` (or `next.config.ts` headers): `Content-Security-Policy`, `Referrer-Policy`, `Permissions-Policy`, strict `X-Frame-Options`.

## Phase 5 – DB and infra
- T5.1 Add indexes listed above as `infra/migrations/023_perf_indexes.sql`.
- T5.2 Add Upstash Redis and configure env vars. Implement a small helper `web/lib/server/rateLimit.ts` used in API routes.

## Phase 6 – Tests and budgets
- T6.1 Unit tests for validators, color sanitizer, and API handlers (happy path + limits + error masking).
- T6.2 Add Lighthouse CI or a scripted `pnpm next build && lighthouse` for key pages; store reports.

# D. Acceptance criteria
- Dashboard initial data load uses ≤2 HTTP requests and renders FCP within target on 4G (simulated), with <200KB JS for above-the-fold.
- Orders POST is transactional; no partial orders; P95 latency improved by ≥30% vs baseline under load test.
- Orders and categories GET enforce pagination and max window; large queries do not exceed 1–2s P95.
- No raw `error.message` returned to clients; correlation IDs present in logs.
- No dynamic `<style>` template injection; invalid colors fall back safely.
- DB indexes exist and are used (explain plans show index scans on filters).
